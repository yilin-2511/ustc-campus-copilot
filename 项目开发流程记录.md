## 🔍 架构审阅记录（2026-06-25）

### 审阅结论：整体架构合理，以下为具体建议和风险点

#### ✅ 做得好的地方
1. **分层清晰**：接入层 → 网关层 → 智能体层 → 工具层 → 数据层，职责分明
2. **多智能体分工合理**：Router + 3个专业助手 + Synthesizer 的模式适合校园场景
3. **技术选型务实**：FastAPI + ChromaDB + Streamlit 都是轻量成熟方案
4. **竞品分析到位**：SJTU Agent 和 campus-info-agent 的对比有参考价值
5. **风险意识好**：LLM配额管理、爬虫维护、隐私保护等都有考虑

#### ⚠️ 需要关注的问题

**1. 框架选型存在重叠**
- 文档同时选了 LangChain + CrewAI + LangGraph，三者功能有大量重叠
- 建议：**主选 LangGraph（状态机编排）+ LangChain（工具/RAG基础能力）**，CrewAI 的角色定义概念可以借鉴但不必引入
- 理由：LangGraph 的图状态机天然适合你们的 Router → 多Agent并行 → Synthesizer 流程

**2. RAG 引擎选型待明确**
- LlamaIndex vs LangChain RAG 二选一即可，不必两个都用
- 建议：**LlamaIndex**，它的数据摄入管线（Data Connectors）更适合你们多源信息整合的需求

**3. 数据源获取是最大风险**
- 文档列出了 10+ 个科大平台，但大部分没有公开 API
- 浏览器自动化（Playwright）爬取需要持续维护，且可能触发反爬
- 建议 MVP 阶段优先用**静态知识库**（手动整理的 FAQ、新生手册等），动态数据源逐个接入

**4. 多智能体通信开销**
- 每个用户请求可能触发 3-4 个 Agent 并行调用 LLM，Token 消耗是单 Agent 的 3-4 倍
- 建议：Router 先做轻量意图分类（不用 LLM，用规则/小模型），减少不必要的 Agent 调用

**5. 部署兼容性**
- 学校算力平台（107.ustc.edu.cn）的环境需要提前验证
- Playwright 需要浏览器二进制文件，确认平台是否支持
- ChromaDB 嵌入式模式对文件系统有要求

#### 📊 优先级调整建议

当前文档的 P0/P1/P2 划分合理，但建议微调：

| 优先级 | 功能 | 调整理由 |
|--------|------|----------|
| **P0** | 智能问答(RAG) + 校园导览 | 静态知识库，不依赖外部API，最稳 |
| **P0** | Web UI (Streamlit) | 演示必需 |
| **P1** | 食堂推荐 | 需要食堂数据源，先确认数据可得性 |
| **P1** | 课表查询 | 需要教务系统对接 |
| **P2** | 活动推荐、成绩查询 | 依赖更多外部系统 |

## 🔬 SJTU Agent 深度分析（2026-06-25）

### 源码分析关键发现

#### 1. Prompt 工程设计精良

SJTU Agent 的 SYSTEM_PROMPT（`prompts.py`）有 **500+ 行**，是它最核心的资产。关键设计模式：

```
核心原则：永远先尝试，不主动说不行
  ↓
工具选择策略（按优先级排序的决策树）
  ↓
各场景 SOP（DDL查询、食堂推荐、成绩查询、邮件发送...）
  ↓
何时停止工具调用（防止无限循环）
  ↓
安全规则（高风险操作需确认）
```

**对你们的启示**：
- Prompt 不是越短越好，SJTU 用详细 SOP 大幅提升了工具调用的准确率
- 你们的 Router Agent 的 prompt 可以参考这个"决策树"结构
- "永远先尝试"这个原则值得直接借鉴

#### 2. 工具设计模式

SJTU Agent 有 **50+ 个工具**，按功能模块拆分：

| 模块 | 工具数 | 示例 |
|------|--------|------|
| 学业管理 | ~15 | get_ddls, get_schedule, query_grades, list_canvas_assignments |
| 校园生活 | ~5 | get_canteen_crowd, recommend_canteen, record_dining_choice |
| 信息服务 | ~8 | search_campus, browse_mysjtu, fetch_url, read_shuiyuan_topic |
| 平台接入 | ~8 | setup_telegram, setup_wechat, setup_feishu, setup_qq |
| 扩展能力 | ~5 | add_mcp_server, create_skill, execute_python |

**对你们的启示**：
- 工具按模块拆分文件（`_dining.py`, `_canvas_files.py` 等），结构清晰
- 每个工具都有详细的 `description`（给 LLM 看的），这是工具调用准确率的关键
- `execute_python` 工具很巧妙——给 Agent 动态执行代码的能力，弥补工具不足

#### 3. 食堂推荐系统（直接可参考）

SJTU 的食堂推荐是一个完整的子系统：

```
实时拥挤度 API → 用户历史偏好 → 内置食堂知识库 → 综合评分 → Top-N 推荐
```

- **实时数据**：从 `campuslife.sjtu.edu.cn` API 获取拥挤度
- **用户画像**：记录每次用餐选择，学习口味偏好
- **知识库**：内置食堂百科（楼层、菜品、营业时间）
- **评分算法**：拥挤度 + 偏好匹配 + 历史记录 综合打分

**对你们的启示**：USTC 如果有类似的食堂数据接口，可以直接复用这套设计

#### 4. 多平台 Bot 接入

SJTU 支持 Telegram/飞书/微信/QQ 四个平台，架构设计值得参考：

```python
# 每个 Bot 入口都调用同一个 Agent 核心
def _capture_turn(sess, user_text):
    sess["messages"][0]["content"] = SYSTEM_PROMPT + date_ctx + platform_ctx
    sess["messages"].append({"role": "user", "content": user_text})
    agent._run_one_turn(client, model, sess["messages"])
    return extract_reply(sess)
```

**对你们的启示**：先做一个 Web UI，Bot 接入可以后期加，核心 Agent 逻辑不用改

#### 5. 记忆系统

SJTU 用 ChromaDB 实现了用户记忆：

```python
# 存储记忆
store_memory(user_id, text, metadata)
# 语义搜索相关记忆
search_memory(user_id, query, n=3)
# 构建上下文注入 system prompt
build_memory_context(user_id, current_msg)
```

**对你们的启示**：这个方案轻量且有效，可以直接复用
