## 0️⃣ 问题背景

### 产品愿景
**让每一位新生都能轻松享受科大丰富的校园资源**

### 现状
- 中科大的校园平台已经比较完善，覆盖了教务、图书馆、食堂、活动等多个方面
- 但这些平台**较为分散**，信息分布在不同的系统、公众号、网站中
- 新生等对校园资源**不够熟悉**，不知道有哪些资源可以利用
- 想要了解信息，需要通过**各类繁琐渠道**：翻公众号、查网站、问学长学姐

#### 非正式信息渠道
除了官方平台，大量信息还分散在**非正式渠道**中：

| 渠道类型 | 示例 | 信息内容 | 特点 |
|----------|------|----------|------|
| **QQ 群** | 新生广场、大物实验群、创新实践基地群、【次元计划】交流群 | 课程资料、实验指导、项目信息、经验分享 | 信息密度低、需要翻找 |
| **微信群** | 班级群、社团群、实验室群 | 通知、活动、资料分享 | 信息碎片化、易错过 |
| **论坛** | 瀚海星云、知乎、小红书 | 经验帖、攻略、评价 | 信息质量参差不齐 |
| **口口相传** | 学长学姐、同学 | 实用建议、避坑指南 | 真实但效率低 |

#### 问题核心
- **资源分散**：10+ 个官方平台 + 大量非正式渠道
- **信息孤岛**：平台间数据不互通，需要重复登录
- **使用门槛**：新生不知道有哪些平台，更不知道如何使用
- **体验碎片化**：不同平台界面、操作方式不一致
- **信息密度低**：QQ 群等非正式渠道信息分散，需要大量翻找

### 痛点
- **信息差**：不知道学校提供了哪些资源
- **获取成本高**：需要花时间去各个平台查找
- **使用门槛高**：即使知道有资源，也不清楚如何使用
- **体验碎片化**：不同平台体验不一致，切换成本高

### 我们的价值
- **打破信息差**：让新生快速了解科大提供了哪些资源
- **降低使用门槛**：通过自然语言交互，让获取信息变得简单
- **提升资源利用率**：帮助学生更好地享受和利用校内资源
- **丰富校园生活**：让校园生活更便捷、更丰富

```
用户：我刚到科大，对校园不熟悉
需求：
  - 校园导航
  - 食堂推荐
  - 办事流程（办卡、借书等）
  - 社团介绍
```

```
用户：中午吃什么？
需求：
  - 根据口味推荐
  - 查看今日菜单
  - 查看排队情况
  - 预算控制
```

```
用户：下周要考试了，帮我规划复习
需求：
  - 查看课表和考试安排
```

```
用户：这周末有什么活动？

  - 查看活动列表

```

## 5️⃣ 功能详细设计

### 5.1 智能问答（P0）
**功能描述**：
- 自然语言提问，智能回答
- 支持多轮对话
- 基于 RAG 的知识库问答

**示例对话**：
```
用户：东区图书馆几点关门？
助手：东区图书馆今天开放到 22:00，周末到 21:00。需要我帮你设置提醒吗？

用户：我想吃辣的，预算20块以内
助手：推荐你去东区一食堂二楼的川菜窗口，今天有：
      - 麻婆豆腐 8元
      - 水煮鱼 18元
      - 宫保鸡丁 15元
      需要导航过去吗？
```

**技术实现**：
- [ ] RAG 知识库
- [ ] 意图识别
- [ ] 多轮对话管理

### 5.2 食堂推荐（P0）
**功能描述**：
- 基于口味偏好推荐
- 基于预算筛选
- 基于位置推荐
- 查看今日菜单
- 查看评价

**输入参数**：
- 口味：辣、甜、清淡等
- 预算：10元、20元、30元+
- 位置：东区、西区、中区
- 时间：早餐、午餐、晚餐

**输出内容**：
- 推荐食堂/窗口
- 今日菜品
- 价格
- 评价
- 导航

### 5.3 共享知识库（P0）⭐ 核心创新

**功能描述**：
- 社区驱动的校园知识共享平台
- 用户可以上传、编辑、评价校园生活经验
- 基于 RAG 的智能检索和推荐
- 内容分类：潜规则、实用指南、美食推荐、避坑指南、办事攻略等

**核心价值**：
- 补充官方信息的不足（官方不写的"潜规则"）
- 沉淀学长学姐的经验
- 让新生少走弯路
- 形成社区互助文化

**内容类型**：
| 类型 | 示例 | 来源 |
|------|------|------|
| **潜规则** | "大物实验报告最好手写，老师更喜欢" | 学长学姐经验 |
| **实用指南** | "东区食堂二楼川菜窗口最好吃，推荐麻婆豆腐" | 用户分享 |
| **避坑指南** | "选课别选XX老师的课，给分低" | 学生口碑 |
| **办事攻略** | "办校园卡去一教一楼，带身份证和学生证" | 经验总结 |
| **美食推荐** | "西门外的烧烤店最好吃，人均30" | 用户评价 |
| **学习资源** | "高数复习资料在蜗壳学社XX帖子" | 资源分享 |

**技术实现**：
- [ ] RAG 知识库（向量检索 + 语义搜索）
- [ ] 内容审核机制（防止不当内容）
- [ ] 用户贡献激励（积分、徽章）
- [ ] 内容评价系统（点赞、踩、评论）
- [ ] 内容时效性管理（过期内容标记）

**示例对话**：
```
用户：大物实验有什么要注意的？
助手：根据学长学姐的经验：
      1. 实验报告最好手写，老师更喜欢
      2. 提前看实验指导书，实验时间紧张
      3. 实验数据可以合理修约，但不能造假
      4. 实验课别迟到，老师会记考勤
      需要更详细的攻略吗？

用户：东门附近有什么好吃的？
助手：根据同学们推荐：
      🌟 张三烧烤（人均30）- 评分 4.8/5
         "羊肉串绝了，就是排队久"
      🌟 李记米线（人均15）- 评分 4.5/5
         "过桥米线很正宗，分量足"
      🌟 王记煎饼（人均8）- 评分 4.3/5
         "早餐首选，加肠加蛋才够味"
      需要导航吗？
```

**用户激励**：
- 贡献内容获得积分
- 积分可以兑换权益（如优先推荐、专属标签）
- 优质内容作者获得"校园达人"徽章
- 定期评选"最佳贡献者"

**内容审核**：
- 机器审核：敏感词过滤
- 人工审核：管理员审核争议内容
- 用户举报：社区自治
- 内容时效性：过期内容自动标记

### 5.4 校园导航与地图服务（P0-P1）

> **2026-06-25 更新**：将地图功能从单纯的"导航"升级为贯穿多个场景的**基础能力**。

**核心理念**：地图不是独立功能，而是嵌入到所有需要"位置"的回答中。

#### 功能场景

| 场景 | 示例对话 | 地图能力 |
|------|---------|----------|
| **位置标注** | "东区图书馆在哪？" | 回复中附带地图标记点 |
| **周边推荐** | "我在东区，附近有什么好吃的？" | 读取用户位置 → 推荐附近食堂 |
| **路线导航** | "从一教到东图怎么走？" | 显示步行路线 |
| **情境感知** | "帮我找个离我最近的打印店" | 用户位置 + POI 距离排序 |
| **活动定位** | "今晚的活动在哪办？" | 活动地点地图标记 |
| **报到引导** | "新生报到去哪里？" | 报到路线 + 关键地点串联 |

#### 地图数据需求

```
校园地图数据层：
├── 建筑物（名称、坐标、照片、楼层）
│   ├── 教学楼：一教、二教、三教、五教...
│   ├── 图书馆：东区馆、西区馆、南区馆
│   ├── 食堂：东区食堂、西区食堂、美食广场...
│   ├── 宿舍：各宿舍楼
│   └── 行政楼：行政楼、校医院...
├── POI 兴趣点
│   ├── 打印店、超市、快递点、ATM
│   ├── 运动场、体育馆
│   └── 校门、公交站
├── 道路网络（步行路径）
└── 区域划分（东区、西区、中区、南区）
```

#### 技术实现

- [ ] **前端地图组件**：Leaflet.js（开源免费）或 高德地图 JS API
- [ ] **校园 GeoJSON 数据**：手动标注建筑物轮廓和 POI 坐标
- [ ] **用户定位**：浏览器 Geolocation API（需用户授权）
- [ ] **路线规划**：基于校园道路网络的简单 A* 寻路，或调用高德步行导航
- [ ] **Agent 集成**：Agent 回复中输出结构化位置数据，前端自动渲染

#### Agent 输出格式设计

```python
# Agent 回复中嵌入位置信息
{
  "text": "东区图书馆在东区中心位置，从一教步行约5分钟。",
  "locations": [
    {
      "name": "东区图书馆",
      "lat": 31.8215, "lng": 117.2783,
      "type": "library",
      "action": "marker"  # marker | route | area
    }
  ],
  "route": {
    "from": {"name": "第一教学楼", "lat": 31.8201, "lng": 117.2790},
    "to": {"name": "东区图书馆", "lat": 31.8215, "lng": 117.2783},
    "walking_time": 5
  }
}
```

#### 用户位置隐私

- 位置仅在浏览器端使用，不上传服务器
- 首次使用需用户明确授权
- 提供手动选择位置的备选方案（下拉选择"我在东区/西区"）

#### 与现有功能的关联

| 现有功能 | 地图增强 |

## 🔟 核心方向扩展

### 10.1 多智能体协作

**核心理念**：不是用一个 Agent 做所有事，而是让多个专业 Agent 协作完成复杂任务

#### 为什么需要多智能体？
- 校园场景复杂，单一 Agent 难以覆盖所有领域
- 不同场景需要不同的专业知识和工具
- 多智能体可以并行处理，提高效率
- 更符合人类协作的模式

#### 智能体分工
| 智能体 | 职责 | 示例 |
|--------|------|------|
| **Router Agent** | 意图识别、任务分解、智能体分配 | 识别用户想问食堂还是课程 |
| **学习助手** | 课程、考试、自习室、图书馆 | 推荐自习室、查询课表 |
| **生活助手** | 食堂、交通、报修、快递 | 推荐食堂、查询班车 |
| **社交助手** | 活动、社团、交友 | 推荐活动、介绍社团 |
| **Synthesizer Agent** | 结果整合、个性化调整、最终输出 | 整合多个 Agent 的结果 |

#### 协作流程示例
```
用户："帮我规划下周的学习安排"

1. Router Agent 识别意图：学习规划
2. 分解任务：
   - 查询课表 → 学习助手
   - 查询考试安排 → 学习助手
   - 推荐自习室 → 学习助手
   - 检查活动冲突 → 社交助手
3. 各 Agent 并行处理
4. Synthesizer Agent 整合结果：
   - 生成完整的学习计划
   - 考虑用户偏好（如喜欢安静的自习室）
   - 输出最终建议
```

#### 用户价值
- **更专业的回答**：每个 Agent 专注自己的领域
- **更高效的处理**：并行处理多个子任务
- **更完整的方案**：整合多个维度的信息
- **更自然的交互**：用户不需要知道背后有多个 Agent

### 10.2 深度个性化

**核心理念**：不只是记住用户偏好，而是真正"懂"用户

#### 个性化层次
| 层次 | 描述 | 示例 |
|------|------|------|
| **L1 基础偏好** | 用户主动设置的偏好 | "我喜欢吃辣" |
| **L2 行为学习** | 从用户行为中推断偏好 | 经常去东区食堂 → 推荐东区 |
| **L3 情境感知** | 根据时间、地点、天气调整 | 下雨天推荐近处食堂 |
| **L4 主动服务** | 预测用户需求，主动推荐 | 考试周前提醒复习 |

**技术实现建议**：
```python
# HTTP 请求（推荐用于公开页面）
import httpx

async def fetch_library_hours():
    async with httpx.AsyncClient() as client:
        response = await client.get("https://www.lib.ustc.edu.cn/本馆概况/开放时间/")
        return response.text

# Playwright（推荐用于需要登录或 JavaScript 渲染的页面）
from playwright.async_api import async_playwright

async def fetch_classroom_info(date="2026-06-25"):
    async with async_playwright() as p:
        browser = await p.chromium.launch(headless=True)
        page = await browser.new_page()
        await page.goto("https://catalog.ustc.edu.cn/query/classroom")
        results = await page.inner_text("#classroom-list")
        await browser.close()
        return results
```

**反爬策略**：
- 请求频率控制：每 1-2 秒一次请求
- User-Agent 轮换：模拟不同浏览器
- 缓存机制：缓存已采集数据，减少重复请求
- 遵守 robots.txt 和学校数据使用政策
