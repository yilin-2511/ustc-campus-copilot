# 嵌入模型升级记录

> **日期**: 2026-06-25  
> **测试知识库**: 58 条（阈值实验产出）  

## 背景

初始版本使用 ChromaDB 内置的 `DefaultEmbeddingFunction`（all-MiniLM-L6-v2），这是英文模型，对中文的语义区分很差。学校网络屏蔽 HuggingFace，无法直接下载中文模型。

## 解决方案

通过 **ModelScope**（国内模型社区，校内网络可访问）下载 `BAAI/bge-small-zh-v1.5`，保存在本地 `models/` 目录。使用 `chromadb.utils.embedding_functions.SentenceTransformerEmbeddingFunction` 加载本地模型。

## 检索效果对比

| 查询 | all-MiniLM-L6-v2 | bge-small-zh-v1.5 | 提升 |
|------|-----------------|-------------------|------|
| "保研夏令营被拒" | 59.9%（命中） | **75.4%**（命中） | +15.5% |
| "怎么预约机房和计算资源" | 35.2%（偏离） | **72.3%**（命中） | +37.1% |
| "大创项目可以不做了吗" | 38.9%（偏离） | **71.1%**（命中） | +32.2% |

## 结论

bge-small-zh-v1.5 三个查询全部 top-1 命中，英文模型两个完全偏离。**中文检索场景下，专用中文嵌入模型是不可替代的。**

# 阈值对照试验

> **实验日期**: 2026-06-25  
> **数据源**: 求助&答疑 标签前 100 条帖子  
> **模型**: deepseek-v4-pro  
> **目的**: 验证 `MIN_LIKES_FOR_VALUE` 参数（帖子中最佳回复的质量分阈值）能否放宽，在不牺牲质量的前提下提高产出率

---

## 背景

初始 Pipeline 设定 `MIN_LIKES_FOR_VALUE = 2`，即帖子必须有一条回复拿到 ≥2 个赞/被引用才算"有价值"。测试跑完 100 条帖子只产出 7 条知识，**产出率仅 7%**。怀疑阈值太严，想验证降到 ≥1 能否提产。

## 方法

**控制变量法** — 同一批 67 条候选帖子，所有帖子全部送 LLM 合成，再按阈值分组统计。

| 控制项 | 值 |
|--------|-----|
| 数据 | 求助&答疑 前 100 条，回复≥3（67 条） |
| Phase 1-2 | 只拉一次详情 API，三个阈值共用 |
| Phase 3 | 67 条全部送 LLM，按质量分分组评估 |
| 比较阈值 | ≥0 / ≥1 / ≥2 |

## 过程

1. **Phase 1** — 抓取 100 条帖子列表
2. **Phase 2** — 67 条回复≥3 的帖子拉详情，计算每条帖子的 `max_quality_score`（最佳回复的点赞+被引用数）
3. **Phase 3** — 67 条候选全部送 LLM 合成，产出 58 条知识条目，跳过 9 条
4. **分组统计** — 按 `max_quality_score ≥ threshold` 分组，统计各组知识条目数

## 结果

### 按阈值分组

| 阈值 | 候选数 | 知识条目 | LLM 通过率 | 产出率 |
|------|--------|----------|-----------|-------|
| ≥2（旧） | 36 | 35 | 97% | **35%** |
| ≥1（新） | 58 | 52 | 89% | **52%** |
| ≥0 | 67 | 58 | 86% | 58% |

> 注：产出率较首次测试（7%）大幅提升，因为首次测试中 Phase 2 还有一个隐含过滤——只取前 10 条按分数排序的候选，实际送入 LLM 的数量被人为限制。

| ≥2 | 22 | 21 | 95% |

| 6 | 4 | 科大校友求助（医疗纠纷）| 个人事件，LLM 判定不可复用 |
| 7 | 1 | 小组作业赤到大的 | 纯水帖 |
| 8 | 0 | 十五五量子规划 | 问太早，回复只有"同关注" |
| 9 | 1 | 急求环境学院刘军武老师风评 | 回复把名字都搞错了 |

### score=1 被救回的部分条目（示例）

- 少院机房怎么预约？科大有好的计算平台吗？
- 校友如何申请返校参观？
- 计算机学院硕转博淘汰率如何？外院学生转博是否可行？
- 研究生重修刷GPA是否可行？具体流程如何？
- 科大校内是否有规律举办的桌游活动？
- 大二学生如何寻找日常实习？

## 结论

1. **阈值降至 ≥1 是最优选择** — 产出率从 35% 提升到 52%（+50%），score=1 的 LLM 通过率 77%，内容质量与 ≥2 无明显差距
2. **score=1 ≠ 内容差** — 南七茶馆点赞稀疏，有用回复往往只有 1 个赞
3. **score=0 不碰** — 虽然 LLM 通过率也有 87%，但样本质量下降明显（闲聊、无人能答等）
4. **LLM 跳过判断基本准确** — 9 条跳过没有误杀值得提取的知识
5. **存在争议** — 第 6 条（医疗纠纷，score=4）被跳过，回复其实有可复用建议（找医疗律师、上大平台求助），但因提问太个人化被误判 → 已调整提示词

# 提示词改进历程

## v1 — 初始版本（2026-06-25，首次测试用）

```
你是一个校园知识提取助手。请根据以下科大论坛求助帖及其回复，提取一条结构化的知识问答。

## 任务
1. 判断这个帖子是否包含可复用的校园实用知识（如选课建议、保研经验、政策解读、生活攻略）
2. 如果是：提取/合成一段 200-400 字的精炼答案
3. 如果不是（纯宣泄/无结果）：标记为 skip
```

**问题**：判断标准是"帖子是否包含可复用知识"，导致个人化事件（如医疗纠纷、offer 选择）即使回复有可推广建议也被跳过。

**额外修正**：修复了 `response_format={"type":"json_object"}` 导致 DeepSeek API 返回空内容的问题，改为手动从 ````json` 块提取 JSON。

## v2 — 阈值实验后调整（2026-06-25）

```
1. 判断这个帖子的**回复中是否包含了可操作的校园实用信息**
   - 即使提问是个人的、具体的，只要回复给出了可推广到其他学生的方法或策略，就应当提取为知识。
   - 重点看答案能不能帮到后来人，而不是问题本身是否"通用"。
2. question 字段改为：看回答能帮哪些人，提炼成通用情境
3. skip 条件收窄：纯宣泄/无人回答实质性内容/纯灌水
```

**改进点**：
- 判断视角从"帖子"转向"回复"——不因提问个人化而跳过
- question 要求提炼通用情境，而非复述原标题
- skip 条件更精确

## v3 — 支持信息帖（2026-06-25，新增 3 个标签后调整）

```
1. 帖子分为两种类型：
   - 类型A（求助帖）：标题提问，从回复提取答案
   - 类型B（信息帖）：标题/正文直接发布有用信息（通知、攻略、教程），
     将正文提炼为问答。
2. 新增 post_type 输出字段（Q&A | 信息发布）
3. 分类新增 "通知/公告"、"技术/教程"、"求职/实习"
```

**改进点**：支持信息帖，适应新增标签

## v4 — 答案自适应 + references + is_temporary（2026-06-25）

```
1. 答案长度按信息量自适应，不再一刀切 200-400 字
2. 新增 references 字段：提取原文中的网址、邮箱、电话、QQ群号
3. 新增 is_temporary 字段：区分临时通知 vs 永久知识
```

**改进点**：保留硬信息、标注时效性、防止注水

## v5 ✅ 当前版本 — Few-shot prompting（2026-06-25）

```
在 prompt 中加入了 3 个完整输入→输出范例：
  示例1：类型A 求助帖（有回复、永久知识、无references）
  示例2：类型B 信息帖（无回复、临时信息、有references）

**改进点**：
- 零样本 → 小样本，输出格式稳定性大幅提升

- 减少 JSON 解析失败的概率

# 去重机制

> **日期**: 2026-06-25  
> **阈值**: 0.88（余弦相似度）  

## 实现

写入 ChromaDB 前进行两步去重：

| 步骤 | 检查方式 | 效果 |
|------|---------|------|
| ID 去重 | 检查 `entry_id` 是否已在集合中 | 同一帖子不重复写入 |
| 语义去重 | 新条目 embedding 与库中最相似条目比较，≥88% 相似度则跳过 | 换标题重发的帖子、不同标签覆盖的同内容帖子被拦截 |

## 验证

- 重建（--rebuild）：58 条一次性写入，无冗余检查
- 增量（再跑一次）：ID 去重拦住全部 58 条，输出 "无新数据需要写入"
- 语义去重：构造与 n7-2967（夏令营被刷）语义几乎相同的假条目，相似度 85%，阈值 0.88 判定为不同（合理——内容虽相似但改写幅度大）

## 参数

| 参数 | 值 | 说明 |
|------|-----|------|
| `DEDUP_SIMILARITY_THRESHOLD` | 0.88 | 余弦相似度阈值，超过视为重复 |

# 未来修改方向

## 回复选择策略优化（当前痛点）

> **问题**：`MAX_REPLIES_PER_POST=6` + 纯按点赞排序，导致"高赞但空洞"回复占用名额，低赞但信息密度高的回复被截断
> **讨论帖**："想知道普通的科大毕业生都怎么样了"（27 回复 → 只取 6 条，疆场/lidang 等建议被截）

### 方案 A：MMR（最大边际相关性）⭐⭐⭐
与传统信息检索的 MMR 思路一致，选回复时平衡质量和多样性：
1. 按质量分排序，取第一条
2. 后续候选：`score = λ × quality_score - (1-λ) × max_similarity_to_already_selected`
3. 避免 6 条回复都在说同一件事

### 方案 B：两轮 LLM ⭐⭐
1. 第一轮用便宜模型（如 qwen-chat）对每条回复打分："这条回复对解答原文有多大帮助？0-5"
2. 按分数排序，取前 N 条
3. 第二轮用完整 few-shot prompt 合成
4. 成本：额外一轮 LLM 调用，但便宜模型成本低

### 方案 C：实体密度打分 ⭐
1. 统计每条回复中 URL、地名、人名、数字、动词的密度
2. "疆场是科大校友开的餐馆" → 4 个实体 → 高分
3. "确实" → 0 个实体 → 低分
4. 与点赞分加权合并作为最终排序依据
